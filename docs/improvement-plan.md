# Navigation ‚Äî Improvement & Growth Plan
> Package: capell-app/navigation ¬∑ Kind: package ¬∑ Tier: free ¬∑ Product group: Capell Foundation ¬∑ Bundle: foundation ¬∑ Status: Draft

## 1. Snapshot

Navigation owns editable, site- and language-scoped menu trees stored as a JSON `items` column on the `navigations` table (one model, `src/Models/Navigation.php`), plus the page-level "which menus reference this page" panel, sync/replication actions, and the foundation header render hook. Surfaces are `admin`, `frontend`, `console`. Key Actions: `BuildNavigationRenderModelAction` (view model assembly + per-request memoization), `BuildPageNavigationReferencesAction` (reverse lookup), `AddPageToNavigationAction` / `RemovePageFromNavigationAction` (item mutation with row locking), `ReplicateSiteNavigationsAction` (site clone). Frontend rendering flows through `View/Components/Header/MainNavigation` and `View/Components/Menu`, which call the build action and `NavigationLoader::getNavigation()` directly in `render()`, then pass a hydrated `NavigationRenderData` into Blade. Dependencies (composer): `capell-app/admin`, `capell-app/frontend` only ‚Äî `capell-app/core` is declared in `capell.json` `requires` but **not** in `composer.json`. Current marketplace summary (verbatim): "Navigation adds site and language scoped navigation trees, page navigation fields, sync actions, and frontend loading support." Manifest declares **1** marketplace screenshot (`docs/assets/marketplace/extension-card.jpg`), but the repo ships **10** dark/light screenshot PNGs under `docs/screenshots/` and `docs/screenshots.json` lists 5 planned surfaces ‚Äî a media/manifest mismatch.

## 2. Improvements (existing functionality)

- **Replace the `LIKE '%...%'` JSON scan in the page-references lookup** ‚Äî `BuildPageNavigationReferencesAction::whereItemsMightContainRecord()` builds `orWhere('items', 'like', '%"pageable_id":123%')` fragments against the JSON text column, an unindexable full-table scan that runs every time the page edit form renders the navigations panel. On MySQL 8 / MariaDB, switch to `whereJsonContains` or a generated/virtual column + index, or maintain a `navigation_page` pivot written by the add/remove actions (which would also kill the LIKE entirely). ‚Äî `src/Actions/BuildPageNavigationReferencesAction.php`, consumed by `resources/views/components/page/navigations.blade.php` ‚Äî M
- **Move DB work out of `render()` in the frontend view components** ‚Äî both `View/Components/Menu::render()` and `View/Components/Header/MainNavigation::render()` call `NavigationLoader::getNavigation()` and `BuildNavigationRenderModelAction::run()` inside the render method. Capell's rule is that the public render path passes hydrated data in. The Blade (`menu.blade.php`, `menu-items.blade.php`) is correctly DB-clean, but the component class itself queries lazily at render time. Resolve the navigation + render model in a frontend resolver / view-composer keyed off the already-resolved `Frontend::site()/page()/language()` and inject `NavigationRenderData`, so rendering is pure. ‚Äî `src/View/Components/Menu.php`, `src/View/Components/Header/MainNavigation.php` ‚Äî M
- **Make the declared health check actually check something** ‚Äî `NavigationHealthCheck` only implements `compatibleCapellApiVersion()` and contains zero health logic, yet `capell.json` registers it with `severity: critical`. A critical check that always passes is worse than none ‚Äî it gives false assurance in Diagnostics. Implement real probes: navigations table exists, at least one navigation per site has the `main` handle, no orphaned `pageable_id` references, render hook registered. ‚Äî `src/Health/NavigationHealthCheck.php`, `capell.json` `healthChecks[]` ‚Äî M
- **Populate the empty manifest fields that downstream tooling reads** ‚Äî `capabilities: []`, `cacheSafety.invalidationSources: []`, and `commands.doctor: null` are all empty/absent. The package clearly has capabilities (menu-tree management, page-navigation-field, navigation-render-model, site-scoped-replication) and the `NavigationObserver` clearly has invalidation sources (`navigation.saved/deleted/restored`, `PageUrlChanged`, `SiteReplicated`). Empty `capabilities` means nothing can declare a dependency on this package's contract. ‚Äî `capell.json` ‚Äî S
- **Add `capell-app/core` to composer `require`** ‚Äî the manifest lists it under `requires` and the code imports `Capell\Core\‚Ä¶` everywhere (`Facades\CapellCore`, `Models\Site`, `Contracts\Pageable`), but `composer.json` only requires `admin` and `frontend`. It works today via transitive resolution, but installing navigation against a different admin/frontend version could silently drop core. Make the direct dependency explicit. ‚Äî `composer.json` ‚Äî S
- **Reconcile marketplace screenshots with what ships** ‚Äî declare the 10 existing `docs/screenshots/*.png` (or the 5 surfaces in `screenshots.json`) in `capell.json` `marketplace.screenshots[]` instead of the single `extension-card.jpg`. Foundation packages are the platform's shop window; one preview image undersells a package that already has full light/dark capture coverage. ‚Äî `capell.json`, `docs/screenshots.json` ‚Äî S
- **Drop the `Site` `pageUrl` fallback query inside the item loader's per-type loop** ‚Äî in `NavigationItemsLoader::getPagesByMorphKey()`, when a page's `pageUrl` site/language doesn't match the active `siteDomain`, the loader issues a fresh `SiteDomain::query()->where(...)->first()` per page. Preload site domains once into a keyed map and look up from memory. ‚Äî `src/Support/Loader/NavigationItemsLoader.php` ‚Äî S
- **Cache the reverse-references result for the admin panel** ‚Äî `BuildPageNavigationReferencesAction` runs uncached on every page form render. Even after fixing the LIKE scan, memoize per-request (mirror the `REQUEST_CACHE_KEY` pattern already in `BuildNavigationRenderModelAction`) since the same page form can re-render the relation manager multiple times in one Livewire request. ‚Äî `src/Actions/BuildPageNavigationReferencesAction.php` ‚Äî S

## 3. Missing Features (gaps)

Manifest `capabilities[]` is empty, so there is no advertised contract to measure against ‚Äî every item below is a real gap relative to navigation norms.

- **Role / permission / auth-conditional item visibility** (differentiator) ‚Äî a grep of `src/Support/Loader` and `NavigationItemData` for `auth`/`Gate`/`can(`/`role`/`permission` returns nothing. The only visibility controls are the `is_visible` boolean and the `visible_from`/`visible_until` publish window. There is no "show this item only to logged-in / guests / role X" capability. This is the single highest-value menu differentiator for a CMS and is entirely absent. ‚Äî `src/Support/Loader/NavigationItemsLoader::visibleMenuItems()`, `src/Data/NavigationItemData.php`
- **Breadcrumbs** (table-stakes for a navigation package) ‚Äî grep for `breadcrumb` across `src/` and `resources/` returns nothing. The package computes `active` state for the current page already (`activeMenuItems()`), so the ancestry data exists, but there is no breadcrumb builder action or component. ‚Äî whole package
- **Richer item types and link targets** ‚Äî `NavigationItemType` has only `Link`, `Page`, `Heading`. There is no first-class "external URL" type (external links are just `Link` with a manual `/`-or-URL string, and the active-state logic in `activeMenuItems()` actually `continue`s past any link whose URL doesn't start with `/`), no divider/separator, no "link to any content model" beyond `Pageable`. `NavigationItemTarget` has a single case (`_blank`) ‚Äî no `rel="noopener"`, no `_self`/`_parent`. ‚Äî `src/Enums/NavigationItemType.php`, `src/Enums/NavigationItemTarget.php`
- **Mega-menu / column layout support** ‚Äî items support `children` (dropdowns, rendered in `header/menu/dropdown.blade.php`) but there is no concept of multi-column mega-menu blocks, featured panels, or embedding a component/promo inside a dropdown. The `data.component`/`component_item` meta hints at per-item component override but there is no mega-menu primitive. ‚Äî `resources/views/components/header/menu/dropdown.blade.php`
- **Per-menu / per-handle programmatic registration** ‚Äî handles are a fixed enum (`Main`, `Footer`, `SubFooter`). A theme or package cannot register a new named menu location (e.g. `account-menu`, `mega-header`) without editing this package's enum. A `NavigableRegistry`-style registry exists for *models*, but not for *menu handles*. ‚Äî `src/Enums/NavigationHandle.php`
- **Active-state for child/descendant URL matching** ‚Äî `urlMatches()` does an exact normalized-string equality (`trim($a,'/') === trim($b,'/')`). A parent section page is not marked active when a child URL is the current page unless the child item itself is in the tree. There is no "starts-with / is-ancestor" active mode, which most themes need for section highlighting. ‚Äî `src/Support/Loader/NavigationItemsLoader::urlMatches()`
- **Public render-model cache (cross-request)** ‚Äî `capell.json` sets `cacheSafety.cacheable: false` and the build action only memoizes per-request. Every cold request rebuilds the render model (resolve pages, build active state). A site/locale-scoped cached render model (the manifest already lists `variesBy: ["site","locale"]`) would meaningfully help the 20ms frontend budget. ‚Äî `src/Actions/BuildNavigationRenderModelAction.php`

## 4. Issues / Risks

- **Unindexable JSON `LIKE` scan on the admin page form** ‚Äî `BuildPageNavigationReferencesAction::whereItemsMightContainRecord()` does `orWhere('items','like','%...%')` over the whole `navigations` table. This blows the manifest `adminQueryBudget: 40` reasoning (it is one query but a full scan) and degrades linearly with navigation count. ‚Äî `src/Actions/BuildPageNavigationReferencesAction.php`
- **Stub health check declared `critical`** ‚Äî `NavigationHealthCheck` has no `check()`/probe logic at all; Diagnostics will report navigation as healthy unconditionally. False-green on a critical check is a production risk. ‚Äî `src/Health/NavigationHealthCheck.php`
- **Public render path lazy-loads in component `render()`** ‚Äî Capell forbids the public render path from querying/lazy-loading; the Blade templates are clean, but `Menu::render()` and `MainNavigation::render()` execute `NavigationLoader::getNavigation()` (DB + cache) and `BuildNavigationRenderModelAction::run()` (which runs `NavigationItemsLoader` queries) at render time. The boundary is being honored in spirit (Blade is pure) but violated in the component class. ‚Äî `src/View/Components/Menu.php`, `src/View/Components/Header/MainNavigation.php`
- **`NavigationLoader::getNavigation()` falls back to `whereNull('language_id')` site-wide query then in-PHP sort** ‚Äî when `siteOnlyFallback` is true and no language match is found, it loads *all* published navigations for the site into memory and sorts in PHP. Fine for a handful of menus, but unbounded by design. ‚Äî `src/Support/Loader/NavigationLoader.php`
- **Observer cache invalidation can fan out to every site** ‚Äî `NavigationObserver::clearCache()` merges `Site::query()->pluck('id')` (all sites) into the keys-to-clear set whenever `site_id` is null on either the current or original record. A single global-navigation save invalidates per-site keys for the entire install. Correct, but a thundering-herd risk on large multi-site installs; the manifest sets `queueInvalidation: true` but `invalidationSources: []` is undocumented. ‚Äî `src/Observers/NavigationObserver.php`
- **No cross-request cache safety test for anonymous output** ‚Äî Capell requires tests proving anonymous/non-admin render safety for cache/render changes. There are frontend render tests (`tests/Feature/Frontend/Page/PageNavigationTest.php`, `PagePerformanceNavigationTest.php`) and the boundary arch test (`tests/Arch/NavigationBoundaryTest.php`) only asserts it doesn't import downstream packages and uses strict equality. There is no test asserting the rendered menu HTML leaks no admin internals (model IDs, field paths, `pageable_type`, signed editor URLs) for an anonymous visitor. The admin page panel does emit `NavigationResource::getUrl('edit')` links ‚Äî confirm that Blade (`page/navigations.blade.php`) is never reachable on a public surface. ‚Äî `tests/Arch/NavigationBoundaryTest.php`, `resources/views/components/page/navigations.blade.php`
- **`NavigationItemData` is mutated in place during loading** ‚Äî `NavigationItemsLoader::load()` reassigns `$this->navigation->items` and flips `$item->active = true` on shared data objects; `BuildNavigationRenderModelAction` memoizes by a cache key that includes `spl_object_id` for unsaved navigations. Mutating model state during a read path is a subtle cache-coherence footgun if the same `Navigation` instance is rendered in two contexts. ‚Äî `src/Support/Loader/NavigationItemsLoader.php`
- **i18n**: item-type and target labels are translated (`getLabel()` via `__()`), and the lang file is `resources/lang/en/generic.php`. Coverage looks thin (one `generic.php`); verify the header Blade's hardcoded `aria-label`/`sr-only` strings all route through `__()` (several do, e.g. `capell-navigation::generic.main_navigation`). ‚Äî `resources/lang/en/generic.php`
- **Test gaps**: well-covered ‚Äî all 6 Actions, both loaders, observer, adapters, content-graph extractor, both commands, policy, registry, and Filament resources/relation-manager all have Pest files. Notably **untested**: cross-request public-output safety (above), the `LIKE`-fragment edge cases in `BuildPageNavigationReferencesAction` (e.g. a `pageable_id` substring collision like 12 matching 123 ‚Äî the `%"pageable_id":12%` fragment can false-positive before the in-PHP `navigationContainsRecord` filter rescues it), and active-state for external links. ‚Äî `tests/`

## 5. Marketplace & Positioning

This is a free, foundation-bundle, first-party package ‚Äî correctly positioned: navigation is non-negotiable infrastructure every Capell site needs, so it belongs in the free foundation tier as a platform credibility piece, not a revenue line.

- **Current `summary` critique**: "Navigation adds site and language scoped navigation trees, page navigation fields, sync actions, and frontend loading support." It is accurate but inward-facing and feature-listy ("sync actions", "frontend loading support" mean nothing to a buyer). It leads with mechanics, not outcome.
  - **Improved summary**: "Build and manage multilingual, per-site menus visually ‚Äî link to any page or URL, nest dropdowns, and render them in your theme with one tag. Active-state, publish windows, and site cloning included."
- **Current composer `description` critique**: "Navigation for Capell" ‚Äî placeholder-grade, zero keywords, zero value.
  - **Improved description**: "Site- and language-scoped navigation menus for Capell: visual menu builder, page & link items, nested dropdowns, active-state rendering, publish scheduling, and multi-site replication."
- **free/bundle vs premium**: keep free/foundation. The premium upsell is *not* this package ‚Äî it is a future `navigation-pro` (mega-menus, role-conditional visibility, breadcrumbs-as-a-service, A/B menu variants) that *depends on* this one's contract. That dependency only works if `capabilities[]` is populated (Section 2/4).
- **Screenshot/media gaps**: manifest advertises 1 image; repo has 10 (light+dark for index, create/edit form, site relation manager, page nav tab, frontend output) plus `hero-desktop.jpg`/`hero-mobile.jpg` that are not referenced in the manifest at all. Wire the existing assets into `marketplace.screenshots[]` ‚Äî the work is already done, it just isn't declared.
- **Platform-pitch contribution**: navigation is the proof that Capell's "editors manage structure without touching theme code" promise is real. The README's "Why It Helps Your Capell Workflow" framing is the right pitch; surface it in the marketplace summary, not just the README.
- **Keywords/tags (8‚Äì12)**: `navigation`, `menus`, `menu-builder`, `multilingual`, `multi-site`, `dropdown`, `mega-menu`, `breadcrumbs`, `active-state`, `cms`, `filament`, `foundation`.
